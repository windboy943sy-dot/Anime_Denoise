## 0. 統合方針のサマリー

先行調査から見えた重要な事実関係を統合方針の根拠とする。

1. **単一アルゴリズムで全劣化に対応するのは不可能**。BM3D/NLMは構造化アーティファクト（ブロックノイズ、リンギング）に弱く、CNN系（NAFNet等）は学習データ分布外の劣化に弱く、Transformer/拡散系（Restormer, StableSR）は高品質だが低速。→ **劣化の種類ごとに専用処理を割り当て、ルーティングする設計**が必要（後述の劣化分解ルーター）。
2. **実写とアニメは、動き・テクスチャ・意図された「劣化」の3点で根本的に異なる**。実写のオプティカルフローはアニメの誇張動作・2/3コマ打ちで破綻しやすく、逆にアニメのベタ塗り・線画構造は実写向けdenoiserがエッジを破壊しやすい。→ **共通基盤（解析層）は共有しつつ、復元層は完全に別パイプラインとする**。
3. **フィルムグレインは「ノイズ」ではなく意図された質感である場合が多い**（調査レポート3節参照）。除去一辺倒ではなく、グレイン推定・分離・必要に応じた再合成というアプローチが実務水準では重要。
4. **リアルタイム性と品質はトレードオフであり、単一の動作モードでは製品として成立しない**。プレビュー用の軽量パスとレンダリング用の高品質パスを最初から分離設計する。
5. **OFXは1エフェクト＝1プラグインが基本だが、Resolveのノードベースワークフローでは複数プラグインが直列に並ぶことが多い**。解析結果（動きベクトル、劣化マップ等）をプラグイン間で使い回せなければ、性能面で競合製品（Neat Video, Dehancer等）に対して不利になる。→ 共有解析基盤の提案（5章）。

## 1. 共通基盤アーキテクチャ

### 1.1 全体パイプライン図

**設計方針の変更点（重要）**: 当初案ではコンテンツ種別（実写/アニメ/フィルムスキャン/デジタル）を自動分類器で判定する設計としていたが、誤判定時にユーザーが処理結果を見て初めて異常に気づく（かつ原因が分かりにくい）という運用上のリスクが大きいため、**種別判定はユーザーがプルダウンで明示的に選択する構造に変更**する。自動推定ロジック自体は完全に廃棄せず、プルダウンの初期値（サジェスト）を決めるための補助情報としてのみ残す（1.2.1節）。

| 構造・動き解析 | フレーム列、種別パラメータ | Optical Flow場、ブロック動きベクトル、フレーム保持マップ（アニメ時）、線画/エッジマップ | 実写: RAFT系（高精度）またはブロックマッチング（高速）を品質モードで切替。アニメ: フレーム保持検出【独自提案 3.2節】を先行実行し、保持グループ内はフロー計算を省略（動きゼロと確定するため計算コスト削減） |

### 1.2.1 コンテンツ種別パラメータのUI設計（プルダウン方式）

**方針**: 実写/アニメの判定を誤ると復元パイプライン全体（2章 vs 3章）が入れ替わり、結果に致命的な破綻（線画を過度に潰す、アニメ特有のステップ動作を誤って滑らかにする等）が起きる。自動分類の信頼度に処理結果を委ねるのはプロ用途のツールとして許容度が低いため、**最終判断は必ず人間が行う**構造とする。OFXでは `kOfxParamTypeChoice`（プルダウン/コンボボックス）で実装する。

**パラメータ構成（2系統に分離）**:

| パラメータ名 | 型 | 選択肢 | 役割 |
|---|---|---|---|
| `contentType`（映像種別） | Choice（必須選択、プルダウン） | 実写 / アニメ | **パイプラインのハード分岐**。2章（実写）と3章（アニメ）のどちらのモジュール群を使うかを確定する。中間や「Auto」は持たせない（曖昧な混在処理を避けるため） |
| `sourceMedia`（素材メディア） | Choice（プルダウン、初期値は自動推定サジェスト） | デジタル / フィルムスキャン / 自動推定（初期表示のみ・確定後は上書き） | ダスト/傷除去、グレイン管理、セル・線画揺れ補正など**フィルム由来モジュールの既定オン/オフ**を決める。誤っていても致命的破綻はしない（該当モジュールが不要に有効/無効になるだけ）ため、初期値には軽量ヒューリスティック（色数分布・グレイン統計スコア等、既存の劣化プロファイリングの副産物を流用）によるサジェストを許容する |

**UI/UX上の工夫**:
- `contentType` はプラグインGUIの最上部・最も目立つ位置に配置し、初回起動時は未選択状態（明示的にユーザーが選ぶまでデフォルト処理を走らせない、または安全側＝最も保守的な処理＝実写・弱デノイズをデフォルトにする）とする。
- `sourceMedia` の「自動推定」はあくまで初期値のサジェスト機能であり、ユーザーが一度手動選択した場合はクリップ単位でその選択を記憶し、以後自動推定で上書きしない。
- クリップ（メディアプールアイテム）ごとにパラメータ値を保持し、同一クリップを複数箇所で使い回してもタイムライン上で再設定不要になるよう、OFXのパラメータ永続化（クリップ単位のインスタンスデータ）を活用する。
- 将来的に自動推定の精度が実運用で十分と確認できた場合でも、**プルダウンの選択肢自体からは「自動」を主要選択肢として残さず、あくまで人間の最終確認を必須とする**方針を維持する（3.1節・4.2節の設計もこれに合わせて修正済み）。

**下流モジュールへの影響**:
- レシピ選択エンジン（4.2節）は、`contentType` を**確定的なハード制約**として扱い、実写用/アニメ用モジュール群のどちらか一方のみを候補集合とする（両方が混ざることはない）。
- `sourceMedia` は各モジュールの**既定の有効/無効フラグ**としてのみ作用し、ユーザーは個別モジュール単位で上書き可能（例: デジタルアニメだが素材にわずかなセル揺れがある場合、セル/線画揺れ補正だけ手動でオンにする、等）。

### 1.3 Shared Analysis Bus 【独自提案】

**課題**: Resolveのノードエディタでは「Deflicker → Denoise → Sharpen → Super Resolution」のようにプラグインを複数直列に並べる使い方が一般的である。各プラグインが独立にOptical Flowやフレーム保持検出、劣化プロファイリングを再計算すると、GPU時間が重複して浪費される。これは競合製品との性能差別化ポイントになり得る。（なお `contentType`/`sourceMedia` はユーザーが各プラグインで個別に設定する前提のため、同一クリップに複数プラグインを適用する際に選択がずれないよう、UI上で同一クリップの他プラグインインスタンスへ設定値をコピーする補助機能もあわせて検討する。）

**提案**: プラグイン群の裏側で共有される「解析結果キャッシュ」を持つ設計とする。

- 実装方式A（推奨・現実的）: 全復元プラグインを**単一の共有ネイティブライブラリ（DLL/dylib/so）としてリンク**し、プラグインごとに異なるOFXバンドルとして公開する。ライブラリ内部でプロセス内シングルトンのキャッシュ（クリップID＋フレーム範囲＋パラメータハッシュをキーとするLRUキャッシュ）を保持し、同一ホストプロセス内であれば複数プラグインインスタンス間で解析結果を再利用する。DaVinci Resolveは同一プロセス内でOFXプラグインをロードするため、プロセス内共有メモリとしての実装は技術的に妥当性が高い。
- 実装方式B（将来拡張・要検証）: ディスクベースのキャッシュ（一時ファイル、クリップのメディアIDとフレーム範囲をキーとする）。プロジェクトを跨いだ再利用や、レンダーファーム分散処理との相性を考慮する場合に有効だが、I/Oオーバーヘッドとキャッシュ無効化ロジックの複雑さがトレードオフ。
- **注意点（明記）**: これは筆者の設計提案であり、OFX仕様自体はプラグイン間の直接通信を規定していない。実装はホスト（Resolve）のプロセスモデルに依存するため、Blackmagic Design側の非公開実装詳細に触れる可能性があり、Phase 1〜2で小規模PoCによる実証が必須。

## 2. 実写向け復元アルゴリズム設計

### 2.1 劣化分解ルーター（Degradation Decomposition Router）【独自提案】

**背景となる課題**: 調査レポートで確認した通り、BM3D/NLM系はランダムノイズ・グレインの除去に強いが構造化アーティファクト（ブロックノイズ、モスキートノイズ、リンギング）には弱い。逆にNAFNet等JPEG artifact除去で学習されたCNNは構造化アーティファクトに強いが、フィルムグレインのような自然な確率過程には過剰適応（グレインを完全に消し「プラスチック的」な質感になる）しやすい。**単一のdenoiserに全劣化を押し付けるのではなく、劣化の統計的性質ごとに専用モジュールへルーティングする**設計とする。

**分解する劣化カテゴリ（本設計での分類軸）**:

| カテゴリ | 統計的特徴 | 担当モジュール |
|---|---|---|
| ランダムセンサーノイズ | 独立同分布に近い、輝度依存（ショットノイズ） | Guided Filter前処理 + 軽量CNN（DnCNN/NAFNet系）残差除去 |
| フィルムグレイン | 空間的に相関を持つ確率的テクスチャ、周波数特性が既知（フィルム感度により分布が変わる） | グレイン推定・分離（後述）＋必要に応じ再合成 |
| 圧縮ノイズ（ブロック/モスキート/リンギング） | 8×8等のブロック境界に整列、エッジ周辺に局在 | ブロック格子検出＋Guided Filterのエッジ適応平滑化＋JPEG/圧縮特化学習モデル（NAFNet系）を境界近傍にのみ適用 |
| ダスト・傷 | 時間的に非連続（1〜数フレームのみ出現）、空間的に局所高輝度/低輝度の線・点 | 動き補償時間外れ値検出＋インペインティング（3節参照の技術を実写にも転用） |
| ブラー（動き/デフォーカス/光学） | 動きブラー: フロー方向に整列した劣化核。デフォーカス: 等方性劣化核 | フロー由来ブラー核推定＋非ブラインドデコンボリューション（動きブラー）／Restormer系デフォーカス専用分岐 |
| 色収差 | チャンネル間の幾何学的ズレ（放射方向） | 古典的レンズモデルに基づくチャンネル別幾何補正（最も安価な処理、パイプライン最初期に実行） |
| フリッカー | フレーム間の輝度・色のグローバル/ローカルな変動 | 時間窓ヒストグラムマッチング＋ロバスト統計（中央値ベース）によるゲイン正規化 |

**擬似コード（劣化分解ルーターの中核ロジック）**:

```
function DegradationDecompositionRouter(frame, neighborFrames, motionField):
    // 1. 各劣化タイプの存在度をスコアリング（0.0〜1.0）
    scores.chromaticAberration = EstimateCAScore(frame)          // チャンネル間ズレ量
    scores.blockArtifact       = EstimateBlockGridScore(frame)   // 8x8/16x16格子相関
    scores.grain               = EstimateGrainScore(frame, neighborFrames) // 高周波の時間非相関成分
    scores.sensorNoise         = EstimateNoiseFloor(frame) - scores.grain // 残差成分
    scores.dustScratch         = DetectTemporalOutliers(frame, neighborFrames, motionField)
    scores.motionBlur          = EstimateBlurKernelFromFlow(motionField)
    scores.flicker             = EstimateTemporalLuminanceVariance(frame, neighborFrames)

    // 2. スコアに応じてモジュールを有効化し、強度パラメータを決定
    recipe = EmptyRecipe()
    if scores.chromaticAberration > THRESH_CA:
        recipe.add(Module.ChromaticAberrationCorrect, strength=scores.chromaticAberration)
    if scores.flicker > THRESH_FLICKER:
        recipe.add(Module.FlickerNormalize, strength=scores.flicker)
    if scores.dustScratch > THRESH_DUST:
        recipe.add(Module.DustScratchInpaint, mask=scores.dustScratchMask)
    if scores.blockArtifact > THRESH_BLOCK:
        recipe.add(Module.Deblock, strength=scores.blockArtifact)
    if scores.grain > THRESH_GRAIN:
        recipe.add(Module.GrainAwareDenoise, grainProfile=EstimateGrainSpectrum(frame))
    else if scores.sensorNoise > THRESH_NOISE:
        recipe.add(Module.StandardDenoise, strength=scores.sensorNoise)
    if scores.motionBlur > THRESH_BLUR:
        recipe.add(Module.MotionAwareDeblur, kernel=scores.blurKernel)

    return recipe  // 実行順序は事前定義された固定順（下記2.2参照）
```

各スコア推定関数（`EstimateGrainScore` 等）は、古典的な統計量（局所分散、周波数帯域エネルギー分布、時間相関係数）の組み合わせで実装可能であり、AI推論を必要としないため、この解析自体は軽量に実行できる設計とする。

### 2.2 実写復元パイプラインの実行順序と各処理の統合設計

実行順序は「幾何補正 → 時間安定化 → 局所欠陥修復 → ノイズ除去 → アーティファクト除去 → ブラー補正 → 超解像 → シャープ/ディテール → 色復元」を基本とする。順序の根拠: 幾何・時間ずれを先に補正しないと後段の動き補償系処理（時間denoise, dust検出）の精度が落ちるため。

| # | 処理 | 統合するアルゴリズム | 設計上の工夫 |
|---|---|---|---|
| 1 | 色収差補正 | 古典的放射方向幾何モデル | 最安価。GPU上でチャンネル別ワープとして実装、以降の全処理の前提を整える |
| 2 | フリッカー除去 | 時間窓（例: 前後5〜9フレーム）のロバスト統計（中央値/切断平均）によるヒストグラムマッチング | 単純な全体ゲイン補正に加え、ブロック単位のローカルフリッカーにも対応するため、Guided Filterでブロック補正マップを平滑化してから適用（ブロック境界の不連続を回避） |
| 3 | ダスト・傷除去 | 動き補償時間外れ値検出＋インペインティング（部分畳み込み系 or 古典的Navier-Stokes/Telea法の高速版をリアルタイムプレビュー用に併設） | 検出マスクの誤検出（動体の一部を傷と誤認）を防ぐため、動きベクトルで補償した上での「時間的に孤立した」外れ値のみを対象とする。品質モードではAIインペインティング、プレビューモードでは古典的手法にフォールバック |
| 4 | グレイン認識型デノイズ | 【統合提案】BM3D/VBM3D（グレイン分離用）＋ NAFNet系（構造ノイズ除去用）のハイブリッド。グレインは完全除去せず「グレインプロファイル」として分離保持し、処理後に強度可変で再合成するオプションを持つ | フィルムルック維持がプロ用途で重要という調査結果を反映。グレイン再合成は疑似乱数ではなく抽出済みグレインスペクトルからの合成とすることで自然な質感を維持 |
| 5 | デブロック/デアーティファクト | ブロック格子検出＋Guided Filterのエッジ適応平滑化＋境界近傍限定でのCNN適用 | 圧縮アーティファクトは全画面ではなく境界近傍のみに存在するため、処理範囲をマスクで限定しGPU負荷を削減 |
| 6 | 動き適応デブラー | RAFT系フロー由来のブラー核推定＋非ブラインドデコンボリューション（動きブラー）／Restormer系デフォーカス分岐 | ブラー種別（動き/デフォーカス/光学的softness）を分岐させ、誤ったブラーモデルの適用による画質劣化を回避 |
| 7 | 超解像 | BasicVSR++系（リアルタイム〜準リアルタイム）／StableSR系（オフライン最高品質） | Render Scale・品質スライダーに応じて自動切替（5章参照） |
| 8 | シャープ／ディテール復元 | Guided Filterベースのアンシャープマスク（コンテンツ適応、残留ノイズを増幅しない） | ステップ4で分離したグレインプロファイルの再合成をこの段階で行うことで「デノイズ→シャープ→グレイン再付加」の順を守り、グレインが二重に平滑化されるのを防ぐ |
| 9 | 色復元 | チャンネル別ヒストグラム補正＋（褪色が激しい場合）Bringing Old Films Back to Life型のキーフレーム誘導色伝播 | フィルムスキャン特有の褪色対応。ユーザーがキーフレームを指定できるUIを想定 |

## 3. アニメ向け復元アルゴリズム設計

### 3.1 実写との根本的な違い（設計前提）

- 動きは**離散的**（2/3/4コマ打ち）であり、連続的なオプティカルフローの前提が崩れる。
- テクスチャが乏しく（ベタ塗り）、パッチベース手法の類似性判定が不安定になりやすい。
- 「ノイズに見えるが意図された表現」（エフェクト作画、フラッシュ、スクリーントーン）が存在し、これを誤って除去すると作品の演出を破壊する。
- 背景（美術）と前景（セル）で動きの性質が異なる（マルチプレーン撮影）。
- フィルムアニメの場合、セル揺れ・線画揺れという実写のカメラブレとは異なる固有のジッターがある。

これらの理由から、**アニメパイプラインは実写パイプラインとモジュールをほぼ共有しない**設計とする（解析層の基盤のみ共有）。どちらのパイプラインを使うかは、1.2.1節で定義した `contentType`（映像種別）プルダウンによってユーザーが明示的に確定する。自動分類には委ねない。

### 3.2 フレーム保持検出（Frame-Hold Detection）【独自提案】

**目的**: 2/3/4コマ打ちにより同一原画が複数フレームにわたって連続する構造を検出し、以降の全処理（デノイズ、フリッカー除去、フレーム補間）に「保持グループ」というメタデータを提供する。

**設計根拠**: 保持グループを認識しないまま処理すると、(a) 時間denoiseがグループ境界で急激な動きを検出しゴーストを生む、(b) フレーム補間が本来ステップ動作であるべき部分を滑らかに補間してしまい「ソープオペラ効果」的な違和感を生む、という2つの重大な問題が生じる。

**擬似コード**:

```
function DetectFrameHolds(frameSequence):
    holdGroups = []
    currentGroup = [frameSequence[0]]

    for i in 1..len(frameSequence)-1:
        prev = frameSequence[i-1]
        curr = frameSequence[i]
        // 知覚ハッシュ + 構造的類似度で「実質同一フレーム」を判定
        // 圧縮ノイズ由来の微小差分を許容するため閾値は劣化プロファイルから動的決定
        similarity = ComputeSSIM(prev, curr, edgeWeighted=true)
        phashDist   = PerceptualHashDistance(prev, curr)

        if similarity > SIM_THRESH and phashDist < HASH_THRESH:
            currentGroup.append(curr)
        else:
            holdGroups.append(currentGroup)
            currentGroup = [curr]
    holdGroups.append(currentGroup)

    // 各グループに保持長（1=1コマ打ち相当, 2=2コマ打ち, 3=3コマ打ち...）を付与
    for group in holdGroups:
        group.holdLength = len(group)
        group.isKeyDrawing = true  // グループの代表フレーム

    return holdGroups
```

**下流モジュールでの利用**:
- **時間デノイズ**: 保持グループ内は「真の静止」として扱い、グループ全体を安全に時間平均できる（動き補償不要、最高品質・最低コスト）。グループ境界を跨ぐ処理では通常のモーション解析にフォールバック。
- **フレーム補間**: デフォルトでは保持グループの「代表フレーム（キー原画）」間でのみ補間を計算し、補間結果を新規中間フレームとして挿入するか、あるいはユーザーが選んだ場合のみ保持を解消して滑らかな動きに変換する「スムージングモード」を明示的オプションとして提供する（意図しない滑らか化を防ぐデフォルト＝原構造維持）。
- **フリッカー除去**: 保持グループ内での輝度変動はほぼ全て圧縮/スキャン起因のノイズと確定でき、フリッカー除去の信頼度を大幅に上げられる。

### 3.3 セル画・線画揺れ補正（Cel/Line Wobble Stabilization）【独自提案】

**目的**: 撮影台のジッターや紙原稿の位置ずれに由来する、フレーム間の微小な位置ズレ（セル揺れ・線画揺れ）を、意図された動き（キャラクターの演技）と分離して除去する。

**設計根拠**: 一般的な映像スタビライザーは特徴点追跡（コーナー検出等）に依存するが、ベタ塗り領域では特徴点が乏しく、線画のみが安定した追跡対象になる。したがって特徴点ベースではなく**線画のスケルトン（骨格線）相関に基づく専用スタビライザー**を設計する。

**擬似コード**:

```
function StabilizeCelWobble(holdGroup):
    // 1. 各フレームから線画のみを抽出（XDoG等のエッジ抽出＋細線化）
    skeletons = [ExtractLineSkeleton(f) for f in holdGroup.frames]

    // 2. グループ代表（最初のフレーム）を基準に、残りのフレームとの
    //    位置ズレを線画相関（正規化相互相関 or Chamfer距離最小化）で推定
    reference = skeletons[0]
    rawOffsets = [EstimateOffset(reference, s, method="line_correlation") for s in skeletons[1:]]

    // 3. 意図的な動き（演技によるパン等）と非意図的なジッターを分離するため、
    //    オフセット系列を低周波（意図的トレンド）と高周波（ジッター残差）に分解
    trend, jitterResidual = DecomposeTrajectory(rawOffsets, method="smoothing_spline")

    // 4. 高周波成分（ジッター）のみを打ち消す補正を適用し、
    //    意図的な動き（トレンド）は保持する
    correctedFrames = []
    for i, f in enumerate(holdGroup.frames[1:]):
        correction = -jitterResidual[i]
        correctedFrames.append(ApplySubpixelWarp(f, correction))

    return [holdGroup.frames[0]] + correctedFrames
```

**注意点**: 本モジュールはセル撮影・紙原稿由来のフィルムアニメで最も効果を発揮する。デジタル制作されたアニメ（レイヤーがピクセル単位で固定）には基本的に不要のため、1.2.1節で定義した `sourceMedia` プルダウン（フィルムスキャン/デジタル）の選択に応じて既定のオン/オフが切り替わる設計とする。ただし最終的な有効/無効はユーザーが個別に上書き可能。

### 3.4 アニメモーション分類（静止／パン／キー動作／エフェクト）【独自提案】

**目的**: 領域単位・フレーム単位で「時間フィルタを適用してよい部分」と「適用してはいけない部分（意図的な高周波表現）」を分離する。

| 分類 | 特徴 | 復元方針 |
|---|---|---|
| 静止（保持グループ内） | フレーム間差分がほぼゼロ | 最強の時間平均を適用可（3.2節） |
| 背景パン/ズーム（マルチプレーン） | グローバルにアフィン近似できる滑らかな動き | 実写的な動き補償型処理（motion-compensated SR等）を適用可 |
| キー動作（原画のステップ動作） | 保持グループ境界で不連続にジャンプする動き | 空間denoiseのみ、時間方向処理は保持グループ内に限定 |
| エフェクト作画（スピード線、フラッシュ、トーン） | 高い時間的分散、周辺構造と無相関 | **時間denoise・フリッカー除去の対象から除外**（誤って「ノイズ」と判定され消されるのを防ぐ） |

**分類方式**: 光フロー場のクラスタリング（アフィン一貫性のある領域＝背景パン、それ以外＝前景）と、フレーム保持マップ、局所時間分散のヒートマップを組み合わせたルールベース分類を一次実装とし、将来的に軽量CNN分類器（4クラス分類、入力は小パッチ＋時間窓）へ置き換え可能な設計とする。

### 3.5 マルチプレーン背景/前景分離【独自提案、実写のレイヤー分離研究の応用】

動きのクラスタリング結果（3.4節）を用いて、背景（滑らかな動き）と前景セル（ステップ動作）を疑似的にレイヤー分離し、それぞれに最適な処理を独立に適用する。背景層には実写向けのMC-SR（BasicVSR++系）を、前景セル層には保持グループ認識型の処理を適用したのち、アルファブレンドで再合成する。完全なレイヤー分離は困難（半透明効果、モーションブラーの重なり等）なため、**ハードなセグメンテーションではなく、動き一貫性に基づくソフトウェイトのブレンディング**として実装することを推奨する。

### 3.6 領域認識型デノイズ（線画保護つきセグメント処理）

- 色量子化ベースのフラット領域セグメンテーション（超高速：色数が少ないアニメでは古典的なk-meansやMean-Shiftで十分）を行い、各セグメント内部は強力な平滑化（Guided Filter or NLM）を適用。
- XDoG等で抽出した線画マスクをGuided Filterのガイド画像として使い、線画近傍では平滑化強度を自動的に弱める。
- フィルムスキャン起源のアニメ（グレインが強い）では、実写同様のグレイン分離パスをセグメント内部にのみ適用し、線画境界には適用しない。

### 3.7 アニメ復元パイプラインの実行順序

```
入力 → フレーム保持検出 → sourceMediaパラメータ参照（フィルムスキャン/デジタル、ユーザー選択済み）
     → [フィルムスキャン選択時のみ既定オン] セル/線画揺れ補正
     → アニメモーション分類（静止/パン/キー/エフェクト）
     → マルチプレーン背景前景分離（ソフトウェイト）
     → 領域認識型デノイズ（エフェクト領域は除外）
     → デバンディング（グラデーション部の圧縮アーティファクト対策、古典的Deband技術）
     → [保持グループ境界のみ] 線画保護型シャープ
     → アニメ特化超解像（Real-CUGAN/APISR系）
     → 色復元（褪色補正、キーフレーム誘導）
     → 出力
```

### 3.8 実写手法との統合ポイント（あえて共有する部分）

完全に別パイプラインとしつつも、以下は実写側と共通コンポーネントとして再利用する（保守性重視）:
- 色収差補正（レンズ由来のためデジタル撮影されたセルにも該当し得る）
- ダスト・傷検出（フィルムアニメのスキャン起因のダストは実写フィルムと同一メカニズム）
- 品質評価モジュール（4.1節、指標のみアニメ用に追加項目を持たせる）

## 4. 新規アルゴリズム総括

### 4.1 品質評価による自己最適化ループ【統合提案】

**設計**: 復元後のフレームに対し、以下のNo-Reference（原画不要）指標を計算し、閾値未達の場合はレシピのパラメータを調整して再処理する。

- **NIQE/BRISQUE系の自然画像統計スコア**（実写向け。学習済みの軽量モデルで十分）
- **時間一貫性スコア**（独自定義）: 隣接フレーム間の動き補償後残差のうち、意図された動き分類（4.4節参照）で説明できない成分の割合。フリッカー・ゴーストの残留度を定量化する。
- **エッジ保持率**（アニメ向け）: 処理前後で線画スケルトンのIoUを計算し、線が過剰に消失・ぼやけていないかを評価。

**再処理ループの設計上の制約（実運用上重要）**: 全フレームに対して毎回このループを回すのは非現実的（レンダリング時間が読めなくなる）。したがって以下の運用を提案する。
- **解析パス（プレビュー/初回のみ）**: ショット単位で代表フレームをサンプリングし、品質スコアが最大化されるパラメータをコーディネート降下法や簡易ベイズ最適化で探索し、ショット単位の「最適レシピ」を確定する。
- **本レンダリングパス**: 確定したレシピをショット全体に適用するのみとし、フレーム単位の再評価ループは行わない（速度優先）。
- ユーザー向けには「Auto-Tune（解析）」ボタンとして明示的に提供し、暗黙のブラックボックス処理にしない。

### 4.2 コンテンツ適応型動的アルゴリズム切り替え（レシピ選択エンジン）【統合提案】

1章・2章・3章で述べた各判定の出力を統合し、実行するモジュール列とその強度を決定する中枢コンポーネント。入力は性質の異なる2種類に分かれる点に注意する。

- **ハード制約（人間による確定値）**: `contentType`（実写/アニメ）と `sourceMedia`（デジタル/フィルムスキャン）。1.2.1節の通りプルダウンでユーザーが確定する。エンジンはこれを分類スコアとしてではなく**分岐条件**としてそのまま使用する。
- **ソフト制約（自動計測される連続値）**: 劣化プロファイル（2.1節の各スコア）、モーション分類（3.4節）、フレーム保持マップ（3.2節）。これらは引き続き自動解析で求め、確定したパイプライン（実写 or アニメ）内でのモジュール強度調整にのみ使う。

実装は以下の2段階とする。

- **第一段階（Phase 1〜3で実装）**: 決定木／ルールベース。各スコアに対する閾値と優先順位を明示的にコード化し、デバッグ・チューニングが容易な形にする。
- **第二段階（Phase 4以降、オプション）**: ショット単位の特徴量（分類スコア、劣化スコア、モーション分類の分布）を入力とする軽量な学習済み分類器（決定木アンサンブル程度で十分、ディープラーニングは過剰）に置き換え、ルールベースでは扱いきれない複合劣化ケースの精度を上げる。ルールベース版をフォールバックとして残し、学習モデルの信頼度が低い場合は自動的にルールベースへ切り替える。

### 4.3 マルチアルゴリズム自動選択（アルゴリズムバンク方式）【統合提案】

各処理段階（デノイズ、超解像等）について、複数の実装（例: Guided Filter＝高速・低品質、NLM＝中速・中品質、NAFNet＝低速・高品質）を「アルゴリズムバンク」として保持し、レシピ選択エンジンがコンテンツ・劣化・ユーザーの速度/品質スライダーに応じて動的に選択する。これによりユーザーは個々のアルゴリズム名を意識せず、単一の「Smart」パラメータで結果を制御できる。上級者向けには手動オーバーライド（強制的に特定バックエンドを指定）も提供する。

### 4.4 フレーム保持時間を利用した処理（3.2節の一般化）【独自提案】

3.2節のフレーム保持検出はアニメに限らず、実写でも三脚固定・静止ショットの検出に応用できる。保持（静止）が検出された区間では、通常のモーション補償より遥かに信頼性の高い単純時間平均が使えるため、**「保持検出」を共通基盤の汎用モジュールとし、実写・アニメ両方のレシピから参照可能な設計**とする（1.2節の共通解析層に組み込む）。

| セル/線画揺れ補正 | 低〜中 | 低 | ○ | 保持グループ全体 | 時間方向（グループ長） | スケルトン抽出のコストが主 |
| Optical Flow（RAFT系） | 高 | 高（コリレーションボリューム） | × プレビューでは軽量版必須 | 2（前後1フレーム、複数ペア推定時は増） | 空間全体 | 品質モードのみで使用、プレビューはブロックマッチングに切替 |
| CNNデノイズ（NAFNet系） | 中〜高 | 中 | △ TensorRT最適化で準リアルタイム可 | 1〜数フレーム（モデル依存） | モデルの受容野分 | half precision推論でVRAM半減 |
| Transformer系（SwinIR/Restormer） | 高 | 高 | × オフライン向け | 1 | 大 | 高解像度でタイル分割必須 |
| VSR（BasicVSR++系） | 高 | 高（伝播バッファ） | △ | シーケンス全体推奨（Recurrent） | 時間方向大 | ショット境界を跨がないようGOP的に区切る設計が必要 |
| 拡散モデルSR（StableSR系） | 非常に高 | 非常に高 | ×× 完全オフライン | 1 | 大 | 「最高品質バッチ処理」として別カテゴリのモード提供 |
| 品質評価（No-Reference IQA） | 低 | 低 | ○（解析パスのみ） | サンプリングフレームのみ | なし | 全フレーム実行は不要 |

### 5.2 キャッシュ戦略

- **解析結果キャッシュ**（1.3節Shared Analysis Bus）: Optical Flow、フレーム保持マップ、劣化スコアを、クリップID＋フレーム範囲＋`contentType`/`sourceMedia`パラメータ値＋その他パラメータハッシュをキーにLRUキャッシュ（メモリ上限は設定可能、超過分はディスクへスピル）。`contentType`/`sourceMedia`はユーザー確定値のためキャッシュキーの一部として扱えば十分で、確率的な再分類処理は不要。
- **プレビュースクラブ対応**: ユーザーがタイムラインをスクラブする際に同一フレームへ何度もアクセスするため、直近アクセスフレームの復元結果自体も短期キャッシュする（OFXホストのキャッシュに加え、プラグイン側でも小規模キャッシュを持つことで、ホスト側キャッシュがクリアされるケースに備える）。
- **時間窓依存モジュールのキャッシュ無効化**: あるフレームのパラメータ変更は前後の時間窓を参照する全フレームの再計算を要求するため、変更されたフレーム番号±時間窓幅の範囲のみを無効化する差分キャッシュ管理を行う。

### 5.3 タイル処理とROI/ROD設計

- 空間タイル分割（`kOfxImageEffectPropSupportsTiles`）は、探索窓・受容野を持つモジュール（NLM, BM3D, CNN）では**タイル境界にカーネル半径分のオーバーラップ（ROI拡張）を要求**し、`getRegionOfInterest` でタイル境界外の追加画素を明示的にリクエストする。境界処理を誤ると継ぎ目アーティファクトが発生するため、Phase 1で最初に検証すべき項目とする。
- 超解像モジュールは出力解像度が入力と異なるため `getRegionOfDefinition` で正しい出力サイズを返す実装が必須。Render Scale（プロキシ解像度でのプレビュー）との組み合わせでは、出力解像度計算がRender Scaleの値に依存しないよう注意する（解像度倍率は常にフル解像度基準で計算し、最後にRender Scaleを乗算する設計とする）。
- 時間方向モジュール（VSR, デノイズ）は `getFramesNeeded` で前後フレームを明示的に要求する。ショット境界（カット点）を越えたフレーム参照は誤補間の原因になるため、**ショット検出結果を全時間方向モジュールに共有し、境界を越えるフレーム参照を自動的に打ち切る**設計とする。

### 5.4 マルチスレッド設計

- `kOfxImageEffectPluginRenderThreadSafety` は `eRenderFullySafe` を目標とし、プラグインインスタンス単位ではなくタイル単位でホストのマルチスレッドサポート（Multi-Thread Suite）を活用する。
- AI推論部分（ONNX Runtime/TensorRT）は内部で独自のスレッドプール・ストリームを持つため、ホスト側マルチスレッドとの二重並列化によるスラッシングを避けるよう、GPU推論はシリアライズしCPU側前処理・後処理のみタイル並列化する設計とする。
- Shared Analysis Busのキャッシュはマルチスレッドから同時アクセスされるため、リーダー・ライターロックまたはロックフリー構造（フレーム単位の粒度であれば実装コストは許容範囲）で保護する。

### 5.5 Render Scale対応方針

- 古典的フィルタ（Guided Filter, 色収差補正等）はRender Scaleに自然に対応可能（カーネル半径をスケールに応じて調整するのみ）。
- AIモデルはトレーニング解像度に依存する性質上、低いRender Scaleでの推論品質が保証されない。**Render Scale ≠ 1.0の場合はAIモデルベースの処理を自動的に軽量な古典手法にフォールバックし、フル解像度レンダリング時のみAI処理を有効化する**方針とする（`kOfxImageEffectPropSupportsMultiResolution` を適切に扱い、プレビュー時のユーザー体験と最終出力品質を両立）。

## 6. プラグイン構成の設計比較

| 構成方式 | 概要 | メリット | デメリット |
|---|---|---|---|
| **完全統合型**（単一プラグインに全機能を内包） | Denoise/Deflicker/SR等を1つのOFXプラグイン内部でタブ切り替え | 解析結果の共有が容易（同一プロセス内）。ユーザーが迷わない | Resolveのノードベースワークフロー（個別にオン/オフ、順序入れ替え、コピー）との相性が悪い。1プラグインが肥大化し保守性低下 |
| **完全分離型**（Denoise, Deflicker, SR等を別々のOFXプラグインとして提供） | 各機能を独立したノードとして提供 | ノードベース編集との親和性が高い。段階的な機能リリースがしやすい | 解析結果（動きベクトル等）の重複計算によるパフォーマンス低下（1.3節の課題） |
| **ハイブリッド型（推奨）** | 共有ネイティブライブラリ（解析基盤＋復元モジュール群）を核とし、その上に複数のOFXプラグインフロントエンド（Denoise, Film Restore, Anime Restoration等）を配置。Shared Analysis Busで解析結果を共有 | ノードベース編集との親和性を保ちつつ、性能面の重複計算問題を解消。保守性（コアロジックは1箇所）と製品ラインナップの柔軟性を両立 | 実装の初期設計コストが高い（共有ライブラリのAPI設計を最初に固める必要がある） |

**推奨**: ハイブリッド型。競合製品（Neat Video, Dehancer等）も内部的には共有コアを持ちつつ複数のOFXバンドルとして製品を分けている前例があり、業界標準的なアプローチと整合する。

**想定プラグインラインナップ**:

| プラグイン名 | 主な対象 | 主要モジュール |
|---|---|---|
| Denoise (Universal) | 実写全般 | 劣化分解ルーター＋グレイン認識型デノイズ |
| Film Restore | フィルムスキャン | ダスト/傷除去＋フリッカー除去＋色復元＋グレイン管理 |
| Deblur & Sharpen | 実写 | 動き適応デブラー＋Guided Filterシャープ |
| Super Resolution | 実写全般 | BasicVSR++系／StableSR系切替 |
| Anime Restoration Suite | アニメ全般 | フレーム保持検出＋セル/線画揺れ補正＋アニメモーション分類＋領域認識型デノイズ＋アニメ特化SR |
| Color Restoration | 実写・アニメ共通 | 褪色補正、キーフレーム誘導色伝播 |

## 8. リスクと注意点（明記）

- 本書中の `【独自提案】` は未検証のアイデアであり、実データでのPoCなしに実装フェーズへ進めるべきではない。特にセル/線画揺れ補正とアニメモーション分類は、対象素材（作品ごとの作画スタイル差）によって性能が大きく変動する可能性が高い。
- Shared Analysis Bus（1.3節）は、DaVinci Resolveのプロセスモデルに関する非公開実装詳細に依存する可能性があり、Blackmagic Design側のOFXホスト実装が将来変更された場合に破綻するリスクがある。フォールバック（キャッシュなしでも正しく動作する）設計を必須とする。
- 拡散モデル・大規模Transformerモデルは推論コストが高く、想定ユーザーのGPU環境（VRAM容量）によっては機能を提供できない可能性がある。VRAM要件を機能ごとに明示し、動作要件を満たさない場合は自動的に軽量モードへフォールバックする設計とする。
- OSS参照実装のライセンス（GPL系のmvtools等）は商用製品への直接組み込みを制限する場合があるため、フェーズ2以降で参照実装からのクリーンルーム再実装を計画的に行う。
- No-Reference IQA指標（NIQE等）は必ずしも人間の主観評価と一致しない。最終的な品質基準はプロカラリスト・映像編集者によるブラインドテストで検証する。
