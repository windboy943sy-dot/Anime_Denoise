# 性能監査（2026-07-13）— 結果不変の冗長処理削減

方針：**出力を1ビットも変えずに**冗長計算を削減する。検証は
(a) 代表出力の MD5 完全一致（before/after）、(b) ゴールデンパリティ、で行う。

## 実施した削減（すべて数値同一）

| # | 箇所 | 冗長の内容 | 対策 |

| 1 | hold_detection.cpp | 隣接ペア判定で同一フレームの gray+GaussianBlur を最大8回再計算（diff と blockSSIM が各2枚、フレームは2ペアに登場、ドリフト検査・refine でも再計算） | gray済み画像を受け取る内部ヘルパーに分離し、呼び出し側で遅延キャッシュ |
| 2 | denoise.cpp analyzeHoldGroup | aligned フレームの luma（フル解像度 float cvtColor）をフリッカー推定・残差・統合ガード・端点判定で計3回以上再計算 | 一度だけ計算して使い回す（フリッカー補正時のみ再計算） |
| 3 | motion.cpp estimateGlobalMotion | ECC前ワープのトレランス残差マップを ECCゲート・改善チェック・面積対決で計3回再計算 | 1回計算して共有 |
| 4 | lineArtEdgeMask（XDoG）C++/Python | 成分フィルタが `setTo(255, labels==j)` を成分ごとに実行＝O(成分数×全画素)。線画は数百〜数千成分 | ラベルLUTの一括適用に変更（O(全画素)） |

## 監査で発見したバグ（パリティ逸脱）

**C++ extend.cpp のエッジ除外が Canny のまま**だった（Python は ★2 で XDoG 化済み）。
共有シンボル `lineArtEdgeMask` に一本化して修正。
※これは唯一の「出力が変わる」変更（C++ の第2層のみ、Python 仕様への整合）。
合成ゴールデンでは差が出ないことを確認済み。

## 検証結果

- detect JSON / reference.png / dust mask / motion JSON：**MD5 完全一致**
- フルパイプライン出力（.mov）：メモ化前後で **MD5 完全一致**
- ゴールデンパリティ：全項目合格

## 実測プロファイル（05_3coma_01、82フレーム、full モード）

| 構成 | 時間 | 内訳 |
|---|---|---|
| extend 0 / grain 0 | **34秒**（0.41秒/フレーム） | 検出＋解析＋統合＋エンコード |
| extend 2 / grain 0 | **43秒**（0.53秒/フレーム） | ＋第2層拡張統合（+9秒） |
| extend 2 / grain 0.3 | 202秒 | ＋フレーム毎NLM（**+159秒＝全体の79%**） |

**結論**：実行時間の支配項は Grain Reduction のフレーム毎 NLM。しかも
extend 有効時は実効N≈13でブレンド重み 1/N≈8% しかなく、「8%の寄与のために
4.7倍の実行時間」を払っている。NLMメモ化はグループ内部フレームにのみ有効で、
3コマ素材は端フレームがフェザリングで異なるためヒットしない（出力同一性は
memcmpゲートで保証済み）。

**運用上の推奨**：extend ≥1 のときは Grain Reduction=0 を既定とする
（視覚差は微小、4.7倍高速）。NLM自体の高速化は Phase 8 の GPU 化
（CUDA/Metal 版 NLM）で扱う。

## 実施しなかった項目（結果が変わる/リスクが高いため将来課題）

- **blockSSIM のブロック毎 boxfilter**：全画像一括SSIMへの融合はブロック境界の
  reflect 境界条件が変わり数値が変わるため見送り（検出系の主要ホットスポット）
- **ダスト検出の leave-one-out 中央値 O(n²)**：ソート済みスタックのランク補正で
  O(n log n) 化可能（数値同一）だが実装リスクが高く、n≤34 では効果が限定的
- **motion の ORB 特徴量再計算**：代表フレームは隣接2遷移で2回 detectAndCompute
  される。API 変更（特徴量キャッシュの注入）が必要
- **プラグインのウィンドウ横断ペア指標キャッシュ**：t→t+1 でウィンドウが12/13
  重複するのに検出を毎回やり直す。(tA,tB,幅) キーのペア指標キャッシュで
  検出コストをほぼ排除できるが、ライブラリAPIの変更が必要
- **NLM の逐次呼び出し**（本質コスト）：Grain Reduction はフレーム毎 NLM が支配項。
  品質を変えずに速くするなら OpenCV CUDA 版 NLM への置換（Phase 8 GPU化の範疇）
