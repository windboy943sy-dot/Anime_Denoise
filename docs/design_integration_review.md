## 1. 我々の実測が設計書の提案を「裏付けた」項目

設計書側が【要検証】としている項目のうち、本プロジェクトで既に検証済みのもの：

| 設計書の提案 | 我々の実測結果 |
|---|---|
| 保持検出をパイプライン先頭に固定（2.2節） | 同一設計で実装済み。コマ打ち87.5%達成 |
| 揺れ補正は剛体変換を既定、非剛体は不安定（3.2節） | **強く裏付け**：ECC AFFINE は自由度過多でグレインに適合し真の止めでもRを劣化させることを実測（denoise_method_survey.md 統合安全ガード節）。Euclidean が正解 |
| 境界信頼度スコアで時間平均の重みを滑らかに（3.1b） | confidence＋境界フェザリングで実装済み。さらに「位置合わせ品質による重み低減」も追加済み |
| 領域単位の保持検出【要検証】（3.1c） | **画素単位で実装・検証済み**：モーションガード（口パク保護）＋統合安全ガード（低速ズーム）＋第2層の画素受け入れマスク。設計書の想定（レイヤー分離してから領域別保持判定）より軽量な「画素マスク方式」で同じ目的を達成している |
| 保持グループ内の光フロー計算スキップ（2.2節） | 同一設計。さらに踏み込み、グループ境界でも Farneback は平坦+グレインで幻覚するため**残差ベースに置換**（実測40%偽動き） |
| フレーム保持検出の実写転用（統合4.4節） | 汎用モジュールとして成立することを支持（我々の drift check・dust-robust 統合も汎用） |
| 実装環境の推奨（Python先行→OFX） | 完了済み：Python PoC → C++（ゴールデンパリティ）→ OFX v1.1（DaVinci実機で検証、二重化問題も解消済み） |

## 2. 我々の実測が設計書に「追記すべき知見」を持っている項目

- **ECC の発散問題**：グレインの強い素材で ECC（Euclidean 含む）が誤収束し、
  整っていたフレームを壊す。残差改善ゲート（Phase 2）と統合安全ガード（Phase 3）が必須
- **端点比較の落とし穴**：フィルム局所歪みは端フレームに集中するため、
  ドリフト/統合判定は内側端点で行う
- **指標の罠**：グレインは Sobel 系鮮鋭度・平均残差を支配する。R の品質判定に
  proxy 指標を使う自動化（設計書4.1の品質ループ）は、指標設計を誤ると
  今回我々が踏んだ罠を全部踏む。**線画IoU など構造ベース指標に限定すべき**
- **DaVinci 実機挙動**：時間方向アクセスは±6で完全供給、サムネイル（92x46）にも
  同一エフェクトが適用される（プロキシ判定とサイズ付きキャッシュキーが必須）、
  SMB からのインストールは権限・AppleDouble の罠がある

| ★4 | **線ボイル配慮**（特化3.2） | 我々のウィーブ補正は「フレーム全体の一貫成分」のみ補正しており設計書の推奨と既に一致。ドキュメントに「線ボイルは補正対象外」であることを明記する（演出保護の観点） |

| ★8 | **Auto-Tune（ショット単位の解析パス）**（統合4.1） | 「暗黙のループにせずユーザーの明示操作にする」方針に賛成。cpp_port_design.md の Analyze パス設計と同じ思想なので統合 |

## 4. 保留・注意して扱う提案

- **アニメ超解像（Real-CUGAN/APISR系）・フレーム補間（RIFE/AnimeInterp）**：
  本プロジェクトの当初スコープ（復元・デノイズ）の外。設計書のプラグイン構成案
  （Anime Restore Core を MVP に、SR等は別プラグイン）はその整理として妥当で、
  我々の「AnimeRestore Denoise」が Core の実装に相当する
- **軽量CNN分類器への置き換え**（特化3.3）：ルールベースで大域93.3%到達済み。
  アノテーションコストに見合う残課題は現状マルチプレーン2件のみ → ★1が先
- **品質自己最適化ループ**（統合4.1）：2章の「指標の罠」を踏まえ、
  導入は線画IoU等の構造指標が揃ってから
- **VBM4D級の重量デノイザのバンク投入**：保持グループ構造がある限り
  第1〜2層で十分な実測結果。動き非依存の素材（実写転用）でのみ検討
