改善後の誤りは、多層セル視差の pan→zoom（05_3coma_01・09_Zentai_01）の2件のみ。
static は 16/16 全問正解。評価は `run_full_evaluation.py`＋`shot_labels_v0.3.csv` で再現可能

誤り分析に基づき実装した改善：

1. **累積ドリフト検査**（`split_drifting_groups`、run_detection 既定有効）：
   グループ端点フレーム同士を直接比較し、累積で動いていたら再帰二分割。
   超低速ズーム（02_Zoom_02：263フレームが1グループに融合→時間統合で
   ゴースト化する危険）が 118 グループに正しく分割されるようになった
2. **ワープ改善チェック**（estimate_global_motion 内）：
   推定ワープと identity を「改善した画素数 vs 新たに壊した画素数」で対決させ、
   壊す方が多ければ static に倒す。静止カメラ＋大面積キャラ芝居で
   ORB がキャラに乗って zoom/rotation と誤答する主因を解消。
   ※平均残差の比較はグレインが支配して機能しない（実測）。画素数対決が正解
3. **ECC 採用ゲート**：ECC（フルアフィン）はグレインの強いパン素材で発散して
   ワープを壊すことがある（実測：01_Pan で発生）。残差が実際に改善した場合のみ
   ECC 結果を採用するようにした
4. **カメラシェイク判定**（`analyze_camera_path`）：並進の方向一貫性
   （circular statistics の R）が低い場合はシェイク＝static 扱い。
   21_Blar（フィックス＋シェイク）が pan 誤判定 → static になった

既知の未解決：多層セルの視差（05_3coma_01・09_Zentai）で pan → zoom 誤り。
単一アフィンでは原理的に曖昧なため、レイヤー分離が必要（優先度は要相談）。

## Phase 2（動き分類）：プロトタイプ実装完了

新規ファイル：

- `prototype/motion_classification/core.py`
- `prototype/run_motion_classification.py`（Phase 1 の JSON を入力にする CLI）

### 設計上のポイント（実測に基づく決定）

1. **保持グループの代表フレーム間で比較する**（連続フレームではなく）。
   コマ打ちアニメでは動きはグループ間でしか起きないため
2. **ノイズ床はカットごとに較正する**：保持グループ「内」のフレームペアは
   「動きゼロ・グレインのみ」の教師データなので、そこから
   「動きが無くても出る残差の上限」を実測する（設計提案書2章の
   「カット単位で正規化」に対応）
3. **Optical Flow（Farneback）は不採用**：平坦＋グレイン領域で大きなフローを
   幻覚する（実測で画素の 40% が偽動き判定）。なし／局所／全体の分類には
   トレランス付き残差（±1px の位置ずれ許容）を使う。
   動きベクトルが必要になる後段で RAFT 等を再検討
4. **フィルム枠の除外**：枠・パーフォレーションはスキャナ座標で固定なのに
   絵はカメラワークで動くため、大域補正後に枠が巨大な偽残差になる。
   `active_area_crop=0.12` で中央領域のみ評価
5. **分類は ORB、ワープ精密化は ECC と役割分担**：マルチプレーン撮影のカットでは
   ORB（特徴点＝前景基準）と ECC（輝度＝面積の大きい背景基準）で推定が食い違う。
   カメラワーク種別の分類は前景基準の方が人間のラベルと一致しやすい
6. 回転閾値 0.5°：フィルムのゲートウィーブが ±0.3° 程度あるため

  ラベリング時に「global_motion はどのレイヤー基準か」を決めておく必要がある

  そろい次第、評価スクリプトを作って調整する（Phase 2 完了基準：85%）
